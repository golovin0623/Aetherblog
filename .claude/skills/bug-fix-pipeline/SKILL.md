# Bug Fix Pipeline — 批量 Bug 分诊、计划与修复工作流

将一份 bug 清单（安全审计报告、QA 测试报告、Code Review 反馈等）转化为系统化的修复过程。
核心理念：**先全量评估，再统一计划，最后并行执行**，而不是逐个 bug 边看边改。

---

## 阶段零：输入识别

1. 用户会提供一个 bug 清单文档（通常是 Markdown）。识别其中的每个 bug 条目。
2. 提取关键字段：编号、标题、严重等级、涉及文件、描述、建议修复方案。
3. 统计总数，向用户确认理解无误后进入阶段一。

如果文件过大（> 256KB），分段读取，确保所有条目都被识别。

---

## 阶段一：全量分诊 (Triage)

**目标**：为每个 bug 做出"是否需要修复"的判断。

对每一个 bug，评估：

| 判定 | 条件 | 说明 |
|------|------|------|
| **FIX** | 存在真实风险或代码缺陷 | 需要编写修复计划 |
| **SKIP** | 风险可接受 / 已有缓解措施 / 误报 | 记录理由，不修复 |
| **DEFER** | 需要更大设计变更，不适合批量修复 | 记录原因和前置条件 |

### 输出物

在 bug 清单所在目录下创建 `fix-plans/` 子目录，生成 `master-checklist.md`：

```markdown
# Fix Master Checklist

| # | Bug ID | Severity | Priority | Triage | Fix Plan | Reason |
|---|--------|----------|----------|--------|----------|--------|
| 1 | BUG-001 | CRITICAL | P0 | FIX | [plan](./bug-001-xxx.md) | 简要原因 |
| 2 | BUG-002 | LOW | - | SKIP | - | 白名单已缓解 |
```

**优先级划分**：
- **P0**：CRITICAL 或可直接利用的 HIGH → 立即修复
- **P1**：HIGH 或影响面广的 MEDIUM → 尽快修复
- **P2**：MEDIUM → 计划修复
- **P3**：LOW + INFO → 持续改进

分诊结果必须覆盖 100% 的条目，不允许遗漏。

---

## 阶段二：修复计划 (Fix Plans)

为每个标记为 FIX 的 bug 创建独立的修复计划文件。

### 计划文件模板

```markdown
# BUG-XXX: 标题

**Severity**: X | **Priority**: PX | **Complexity**: Low/Medium/High

## Problem
一句话描述问题，包含文件路径和行号。

## Changes Required
### File: `path/to/file.ext`
具体的代码变更描述，包含修改前后的代码片段。

## Verification
1. 具体的验证步骤
2. 预期结果
```

### 代码验证

在写修复计划时，**必须先读取实际代码**确认问题存在。审计报告中的行号可能已过时。
如果实际代码与报告描述不符，在计划中标注差异。

### 合并相关修复

如果多个 bug 指向同一个文件或同一个根因，合并到同一个计划文件中，避免冲突。

---

## 阶段三：执行修复 (Fix Execution)

**前置条件**：所有 FIX 条目的计划文件已创建并通过用户确认。

### 批次划分

按优先级从高到低执行：P0 → P1 → P2 → P3。

### 并行执行规则

- 同时最多 **3 个** 并行 Agent
- 每个 Agent 负责独立的文件/模块，避免文件冲突
- Agent prompt 必须包含：
  - 完整的问题描述和上下文
  - 具体的文件路径和修改方案
  - 不允许 Agent 自行扩展修改范围

### 批次间验证

每完成一个批次（约 5 个修复），执行：

1. **编译检查**：`go build ./...`、`pnpm build` 等
2. **测试检查**：`go test ./...`、`pnpm test` 等
3. **Diff 审查**：`git diff --stat` 确认改动范围合理
4. 更新 `master-checklist.md` 中已完成的条目

### 上下文交接

每处理 **10 个计划** 或 **修复 5 个 bug** 后，执行上下文交接：

1. 更新 `master-checklist.md` 的进度
2. 在 checklist 的 **Context Handoff Log** 中记录：
   - 本轮完成了哪些条目
   - 下一轮应该从哪里开始
   - 当前已知的阻塞或风险
3. 输出简要进度摘要给用户

这确保即使上下文窗口被压缩，也能通过读取 checklist 恢复进度。

---

## 阶段四：验证与核查 (Verification)

每个修复完成后：

1. **代码层验证**：编译通过、测试通过
2. **逻辑验证**：根据修复计划中的 Verification 步骤检查
3. **回归检查**：确认修改没有影响相邻功能
4. **Diff 审查**：检查 Agent 是否引入了多余的变更

如果发现 Agent 的修复引入了新问题（如多余的代码块、不该修改的文件），**立即回滚并重新修复**。

---

## 阶段五：收尾 (Wrap-up)

1. 更新 `master-checklist.md`，标记所有 DONE 条目
2. 输出最终统计：
   - 总计 X 个 bug，修复 Y 个，跳过 Z 个，延后 W 个
   - 涉及 N 个文件，+A/-B 行
   - 编译状态、测试状态
3. 列出所有 DEFER 条目及其原因，供用户后续处理
4. 所有变更未提交，等待用户审查

---

## 关键原则

### 不要急于修复
先理解全貌，再动手。全量分诊 + 全量计划完成后才开始写代码。
提前修复可能导致：遗漏关联 bug、重复修改同一文件、修复方向与整体不一致。

### 验证实际代码
报告中的代码片段和行号可能已过时。修复前必须读取当前代码确认问题。

### 控制修改范围
只修报告中提到的问题。不要顺手重构、不要添加注释、不要"改进"周边代码。
Agent 的修复如果超出计划范围，需要回滚多余部分。

### 保持可追溯
每个修复都能追溯到：哪个 bug → 哪个计划 → 哪个代码变更。
`master-checklist.md` 是唯一的 source of truth。

### 优雅降级
如果某个修复复杂度超出预期（需要数据库迁移、需要跨模块重构），标记为 DEFER 而不是强行修复。
在 checklist 中记录原因和前置条件。
