## 2024-03-01 - [Path Traversal in Golang LocalStorage]
**Vulnerability:** Path Traversal vulnerability found in `apps/server-go/internal/pkg/storage/local.go`. The `Upload` and `Delete` methods joined the `basePath` with unsanitized `key` inputs.
**Learning:** `filepath.Join` resolves `../` tokens, so `filepath.Join("/var/www/uploads", "../../../etc/passwd")` returns `/etc/passwd`. An attacker could specify a file key with multiple `../` tokens to break out of the intended upload directory, leading to arbitrary file writes or deletions on the server system.
**Prevention:** Always use `filepath.Abs` to verify the fully resolved base and target destination paths, and assert that the resulting target destination path relates to the base path using `filepath.Rel(absBase, absDest)`. If the relation path begins with or equals `..`, the path has traversed out of the base directory and should be rejected.

## 2025-04-02 - [Fix SSRF and Path Traversal in AI Proxy Endpoint]
**Vulnerability:** The AI proxy endpoint `/api/v1/admin/providers/*` in `AiHandler.ProxyProviders` blindly appended the wildcard parameter (`c.Param("*")`) to the base API path without validation. This allowed an attacker to use `../` to traverse directories and perform Server-Side Request Forgery (SSRF), sending unauthorized requests to arbitrary endpoints on the internal AI service.
**Learning:** Blindly concatenating user input to form proxy URLs allows path traversal, bypassing intended API routing restrictions. This is a common pattern when building pass-through proxies or API gateways without strictly validating the requested path against the intended base directory.
**Prevention:** Always validate and sanitize wildcard path parameters before appending them to base URLs. Check for path traversal sequences like `..` (e.g., using `strings.Contains(subPath, "..")`) and ideally use path normalization (`path.Clean()`) to ensure the final resolved path strictly resides within the intended directory hierarchy.

## 2025-10-24 - [Bypass of Path Traversal check via URL Encoding in Echo Param]
**Vulnerability:** The previous path traversal check `strings.Contains(subPath, "..")` in `AiHandler.ProxyProviders` was bypassed because Echo's `c.Param("*")` returned URL-encoded sequences (e.g. `%2e%2e` or `%2f`). This allowed an attacker to use URL-encoded dot-dot-slash to traverse directories and perform Server-Side Request Forgery (SSRF).
**Learning:** Echo router parameters (`c.Param`) are not automatically URL-unescaped in all versions or situations. String matching against `..` fails if the dots are represented as `%2e`. Input validation must be performed on the canonical, unescaped form of the user input.
**Prevention:** Always decode parameters using `url.PathUnescape()` before checking for forbidden sequences like `..` or use robust path resolution functions like `path.Clean` after decoding to ensure the path doesn't escape the intended directory.
